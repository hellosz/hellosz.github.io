在 Docker 世界里，容器默认是无状态、可替换的。容器一旦被删除，写在容器文件系统里的数据通常也会一起消失。  
这也是为什么只会 `docker run` 还不够，真正把服务跑稳，迟早都要理解数据卷。

这篇文章专门梳理 Docker 三种常见挂载方式：`Named Volumes`、`Bind Mounts` 和 `tmpfs Mounts`。目标不是把概念堆满，而是帮自己建立一个足够清晰的判断框架。

<!--more-->

## 一、先记住结论

如果只记一句话，可以记这个：

- 程序生成的数据，用 `Named Volume`
- 人工要直接编辑的文件，用 `Bind Mount`
- 不需要落盘的敏感或临时数据，用 `tmpfs`

多数场景下，这个判断已经够用了。

## 二、三种挂载方式对比

| 特性 | 命名卷（Named Volumes） | 绑定挂载（Bind Mounts） | 临时卷（tmpfs Mounts） |
| --- | --- | --- | --- |
| 定义方式 | 在 `docker-compose.yml` 顶部声明卷名，或命令里直接指定名称 | 直接挂载宿主机路径，如 `./src`、`/data/app` | 指定 `tmpfs` 挂载 |
| 存储位置 | Docker 管理的专用目录 | 宿主机指定目录或文件 | 宿主机内存（RAM） |
| 生命周期 | 独立于容器，删容器不等于删卷 | 跟随宿主机文件系统存在 | 容器停止或删除后立即消失 |
| 权限处理 | Docker 统一管理，通常更省心 | 需要自己处理 `chown`、`chmod`、用户映射等问题 | Docker 管理，仅内存访问 |
| 跨平台性 | 高 | 低到中，依赖宿主机路径结构 | 中，通常用于 Linux 容器环境 |
| 性能 | 高 | Linux 上通常很好，Mac/Windows 可能受虚拟化影响 | 最高，内存读写 |
| 安全性 | 高 | 取决于挂载路径，误挂敏感目录风险较大 | 高，数据不落盘 |

## 三、命名卷：最适合程序自己生成的数据

`Named Volume` 由 Docker 守护进程管理。你知道它存在，但通常不需要关心它具体落在哪个目录。

### 1. 示例

```yaml
services:
  db:
    image: mysql:8
    volumes:
      - db_data:/var/lib/mysql

volumes:
  db_data:
```

### 2. 为什么它适合生产数据

数据库、上传文件、缓存快照、搜索索引，这类数据有几个共同点：

- 不是给人直接手工编辑的
- 需要跨容器生命周期保留
- 对权限和稳定性比较敏感

这正是 `Named Volume` 的强项。它把“数据持久化”和“宿主机路径细节”隔开了，减少了很多环境差异问题。

### 3. 典型场景

- MySQL、PostgreSQL、MongoDB 的数据目录
- 应用上传文件目录
- 多个容器共享的一份业务数据
- 不想自己处理宿主机和容器内 UID/GID 冲突时

### 4. 适合它的核心原因

- 删容器不会顺手把数据删掉
- 迁移和备份更标准
- 跨平台更稳定
- 权限问题通常比绑定挂载少

如果是在生产环境跑数据库，默认就该先想到 `Named Volume`，而不是宿主机目录直挂。

## 四、绑定挂载：最适合开发时直接改文件

`Bind Mount` 的本质，是把宿主机的某个文件或目录直接映射到容器里。  
它的优点非常直接：宿主机改了，容器里立刻能看到；容器里改了，宿主机也会同步变化。

### 1. 示例

```yaml
services:
  web:
    image: node:20
    working_dir: /app
    volumes:
      - ./src:/app/src
      - ./config/app.yaml:/app/config/app.yaml:ro
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro
```

### 2. 它为什么适合开发

开发阶段最需要的是“快反馈”：

- 本地改代码，容器内立即生效
- 不必每次都重建镜像
- 配置文件和初始化脚本可以直接注入

这就是为什么前端热更新、后端本地联调、数据库初始化脚本，几乎都离不开 `Bind Mount`。

### 3. 典型场景

- 本地开发时挂载源码目录
- 挂载 `.env`、`config.yaml`、`nginx.conf`
- 挂载初始化 SQL 或启动脚本
- 把日志目录挂到宿主机方便排查

### 4. 它的问题也很明显

- 强依赖宿主机路径结构
- 容易遇到权限问题
- 配错路径会覆盖容器内原文件
- 如果挂了危险目录，容器可能直接修改宿主机关键文件

### 5. 使用建议

- 只在确实需要“直接编辑宿主机文件”时使用
- 配置文件能只读就加 `:ro`
- 不要随手挂 `/`
- 生产环境谨慎使用，除非路径、权限、备份策略都很明确

一句话总结：`Bind Mount` 很适合开发，但它带来的自由，也意味着更多约束和风险需要自己兜底。

## 五、tmpfs：最适合不该落盘的数据

`tmpfs` 挂载把数据放在内存里，而不是硬盘里。  
容器一停，数据就没了。这种“短命”反而正是它的价值。

### 1. 示例

```yaml
services:
  secure-app:
    image: alpine
    tmpfs:
      - /run/secrets
      - /tmp/cache
```

### 2. 典型场景

- 解密后的密钥
- Session Token
- 一次性临时文件
- 高速但允许丢失的缓存
- 只读根文件系统下，应用必须写入 `/tmp` 或 `/run`

### 3. 适合它的原因

- 不落盘，敏感数据更安全
- 速度非常快
- 容器结束后自动清理，不留尾巴

### 4. 它的限制

- 数据不能持久化
- 占用内存
- 不适合长期保存业务数据

所以它不是“更高级的 Volume”，而是“专门解决临时数据问题”的工具。

## 六、如何做选择

每次纠结挂载方式时，我会按下面这套顺序判断。

### 1. 数据需不需要保留

如果不需要保留，而且希望速度快、痕迹少，优先用 `tmpfs`。

如果需要保留，再看下一个问题。

### 2. 数据是谁在修改

如果主要是“人”在宿主机上编辑，比如：

- 源代码
- 配置文件
- SQL 初始化脚本

那就用 `Bind Mount`。

如果主要是“程序”在容器里生成，比如：

- 数据库文件
- 用户上传内容
- 程序运行日志
- 缓存或索引

那就用 `Named Volume`。

### 3. 一张简单决策表

| 问题 | 建议 |
| --- | --- |
| 数据不需要落盘吗？ | 用 `tmpfs` |
| 数据要长期保留，而且主要由程序生成吗？ | 用 `Named Volume` |
| 数据需要开发者在宿主机直接编辑吗？ | 用 `Bind Mount` |

## 七、常用命令速查

### 1. 查看和管理命名卷

```bash
docker volume ls
docker volume inspect <卷名>
docker volume rm <卷名>
docker volume prune
```

### 2. Compose 清理

```bash
docker compose down -v
```

这个命令会停止容器，并删除 Compose 创建的命名卷。  
它不会删除你宿主机上的绑定挂载目录，但会把卷里的数据一起删掉，执行前最好确认。

### 3. 备份命名卷

```bash
docker run --rm \
  -v <卷名>:/source \
  -v $(pwd):/backup \
  alpine \
  tar czf /backup/backup.tar.gz -C /source .
```

这个思路很实用：临时起一个容器，把卷挂进去，再打成 `tar.gz` 导出。

## 八、我自己的实践建议

如果是为了少踩坑，我会这样记：

- 生产环境数据库：优先 `Named Volume`
- 本地代码开发：优先 `Bind Mount`
- 配置文件：开发环境用 `Bind Mount`，生产环境尽量收敛到镜像或配置管理
- 敏感临时数据：优先 `tmpfs`

还有一个很重要的经验：

不要把“能挂进去”当成“应该挂进去”。  
挂载方式本质上是在定义容器和宿主机之间的边界，越随意，后面越难维护。

## 九、总结

Docker 的三种挂载方式，并不是彼此替代，而是各自服务于不同问题：

- `Named Volume` 解决“程序数据要持久化”
- `Bind Mount` 解决“宿主机文件要直接参与容器运行”
- `tmpfs` 解决“数据不能落盘，且只需短暂存在”

只要先分清“数据要不要保留”和“数据由谁编辑”，大部分场景的选择就不会错。
